🦠 Antivirus & rootkit scanning

Host-installed ClamAV + rkhunter + chkrootkit running on every node. One install per host covers every LXC, every Docker container, and the host filesystem (ClamAV reads directly into /var/lib/lxc, /var/lib/vz, /var/lib/docker). ClamAV hits are quarantined (chmod 000 + moved to /var/quarantine/wolfstack/) and any processes using the file are killed. rkhunter / chkrootkit findings are alert-only (false-positive rate is too high for auto-action). VMs are not scanned — their disks are locked while running; that's a separate libguestfs/guest-agent feature.

What is WolfRun?

Think of it as Kubernetes for normal people. WolfRun lets you deploy containers across your cluster without writing YAML files or learning complex orchestration tools. Just pick an image, set how many replicas you want, and WolfRun handles the rest.

Self-healing — crashed containers restart automatically
Scheduling — spreads load across nodes
One-click scaling — add or remove replicas instantly
Docker & LXC — orchestrate both runtimes
Container failover — standby containers auto-promote on node failure

How Container Failover Works

  1. Enable failover on any WolfRun service (deploy modal or settings).
  2. WolfRun automatically pre-creates standby containers on other cluster nodes — stopped but fully configured with the same WolfNet IP allocation.
  3. Every 15 seconds, the cluster leader checks if any node has gone offline.
  4. If a node with active containers goes down, standby containers on surviving nodes are instantly started — no re-imaging, no IP reassignment.
  5. A failover event is logged so you can see exactly what happened and when.

No shared storage needed — use app-level clustering (Galera, GlusterFS, WolfScale replication) for data persistence across nodes.

Automatic Docker Image Updates

                        
Watch upstream registries for newer image digests on your running Docker containers, optionally apply them on a maintenance schedule with backup + health-check + auto-rollback. Per-container overrides on the Docker page.


                        

                            
                            Polls upstream registries on the check interval below. When disabled, no checks or applies happen.
                        


                        

                            
                            
                            How often the watcher queries upstream registries for new digests. Each query is one HEAD request per running container — fast, but rate-limits exist.
                        


                        

                            
                            
                            Per-container overrides on the Docker page take precedence. Pinned policy is per-container only (you have to choose what to pin to).
                        


                        
Maintenance window

Leave blank to apply immediately on detection.

                                 What are passkeys, and how do I use them? — click to expand
                            
                            


                                
What is a passkey?

                                

                                    A passkey is a small cryptographic key stored securely on your device — your laptop’s
                                    Secure Enclave, your phone, or a USB security key (YubiKey, Titan, etc.). Instead of typing
                                    a password, you confirm a fingerprint, face scan, PIN, or button press, and your device
                                    proves to WolfStack that it’s really you. Passkeys can’t be phished, can’t be reused
                                    across sites, and can’t be leaked in a database breach.
                                


                                
How do I add my first one?

                                
    
                                    
  1. Type a label that will help you recognise it later (e.g. “MacBook Touch ID”, “YubiKey”, “iPhone”) and click Add passkey.
    2. 
                                    
  2. Your browser will pop up its built-in passkey dialog. Choose where to save it — this device, your phone (via Bluetooth/QR), or an inserted security key.
    3. 
                                    
  3. Confirm with your fingerprint, face, PIN, or button press.
    4. 
                                    
  4. That’s it. Next time you sign in, click “Sign in with passkey” on the login page and skip the password.
    5. 
                                


                                
Add more than one

                                

                                    We strongly recommend registering at least two passkeys — ideally on different devices
                                    (e.g. your laptop plus your phone, or a primary YubiKey plus a backup). If you lose access
                                    to one, the others still let you in. Without a backup passkey, you’d fall back to your
                                    Linux password — which is fine, but defeats the point of passwordless login.
                                


                                
Important: passkeys are tied to the URL you’re visiting

                                

                                    A passkey registered while visiting WolfStack at https://wolfstack.example.com:8553
                                    won’t work if you later visit the server at its IP address (https://10.0.0.5:8553).
                                    Browsers refuse to do passkeys on bare IP addresses at all. Pick a hostname — ideally a
                                    real DNS name — and stick to it. If you have to use IPs, you’ll need to keep using passwords.
                                


                                
What if I lose my device?

                                

                                    Two safety nets:
                                

                                
    
                                    
  • Other passkeys still work. If you registered more than one, just sign in with another.
    • 
                                    
  • Your Linux password still works. The classic username + password form on the login page is unaffected by passkey registration. Sign in that way and remove the lost device from this list.
    • 
                                


                                
Common questions

                                
    
                                    
  • Do passkeys replace my password? Not in this version. They sit alongside it — you can use either to sign in.
    • 
                                    
  • Does this work in any browser? Any modern browser supports WebAuthn: Chrome, Firefox, Safari, Edge. We don’t support older IE-style browsers.
    • 
                                    
  • Do I need a hardware key? No. Most modern devices have a built-in passkey store (Touch ID on Mac, Windows Hello, Android biometrics, iCloud Keychain). A YubiKey or similar gives you a portable, hardware-backed option but isn’t required.
    • 
                                    
  • Where are passkeys stored? The private half stays on your device — we never see it. WolfStack only stores the public half plus a label, in /etc/wolfstack/webauthn.json (mode 0600).
    • 
                                    
  • Can I use the same passkey across multiple WolfStack servers? Yes — if you register the same authenticator on each server. Each server has its own list; there’s no central sync.
    •

Reverse Proxy

                        

                            If WolfStack sits behind a reverse proxy (Cloudflare Tunnel, nginx, Traefik, Caddy, …), status page links and other shareable URLs default to the current browser origin. That works out of the box for most setups — including custom domains on the main API port. Use the override below only if the auto-detection doesn't match the URL your users actually reach WolfStack on (e.g. subpath proxying, or admin-vs-public split domains).
                        

                        

                            

                                
Proxy checklist

                                
    
                                    
  • Forward the proxy to the main API port (default 8553) — status pages, cluster browser, consoles and the SPA all share this port.
    • 
                                    
  • Pass through these headers: Host, X-Forwarded-Proto, X-Forwarded-For.
    • 
                                    
  • Enable WebSocket upgrades (Upgrade and Connection headers) — consoles, VNC, cluster browser all use WS.
    • 
                                    
  • Port 8550 (dedicated no-auth status listener) does not need forwarding — /status/{slug} is also served on the main port.
    • 
                                

                            

                            
                            
                            

                                Absolute URL, no trailing slash. Leave empty to use the current browser origin (the default, works for most setups).
                                Examples: https://status.example.com, https://example.com/wolfstack.

                            Commit WolfStack's configuration — every /etc/wolfstack/*.json (except the GitHub token itself) plus every docker-compose.yml under /etc/wolfstack/compose/ — as a single atomic commit on the branch you configure. Use the commit history as an audit log of state changes; restore pulls the latest tree back into place.

    3. 
                                    
  3. Generate a token: classic tokens need the repo scope, or use a fine-grained token scoped to this single repo with Contents: Read and write.

What is Threat Intelligence?

                                

                                    WolfStack pulls public lists of known-bad IP addresses (botnets, scanners, ransomware command-and-control, spam sources)
                                    from a few well-respected providers, deduplicates them into a single list, and tells the kernel firewall to
                                    drop traffic to or from any of those IPs in both directions: incoming attackers reaching your services,
                                    AND outgoing connections from your containers/VMs to known C2 servers (the canonical “ransomware calling home” case).
                                    Your existing WolfRouter firewall rules still apply on top — threat intel is one extra layer that runs first.
                                


                                
Recommended first-time workflow

                                
    
                                    
  1. Tick "Enabled" but leave "Dry-run mode" on. Click Refresh now. The system will fetch all enabled feeds and show you exactly what would be blocked — without touching the kernel firewall.
    2. 
                                    
  2. Review the blocklist preview. Look at the sample IPs. Use Lookup IP to spot-check a few addresses you care about (your office IP, a friend’s server, etc.) to make sure they’re not on the list.
    3. 
                                    
  3. If anything important is on the list, add it to the Allowlist and save. Allowlisted CIDRs are stripped from every future refresh.
    4. 
                                    
  4. When you’re happy, untick "Dry-run mode". The kernel rule is installed, the badge flips to ENFORCING, and bad IPs start getting dropped.
    5. 
                                


                                
The feed providers explained

                                
    
                                    
  • Spamhaus DROP / EDROP / dropv6 — Hijacked IP ranges and direct allocations to criminal organisations. Very high quality, very low false-positive rate. No API key needed. Recommended for everyone.
    • 
                                    
  • FireHOL Level 1 — A curated union of the most consistent “definitely-attacking-someone-right-now” feeds. Larger than Spamhaus but still well-maintained. No API key needed. Recommended for everyone.
    • 
                                    
  • CrowdSec community blocklist — Real-time IPs being reported by the global CrowdSec community. Fresher than Spamhaus but with a bit more noise. API key required — sign up free at app.crowdsec.net.
    • 
                                    
  • AbuseIPDB — A long-running community-curated database. We pull only entries with a 90%+ confidence score. API key required — free tier at abuseipdb.com gives 1000 requests/day, plenty for our 6-hour refresh.
    • 
                                

                                

                                    Enabling more providers makes the blocklist bigger but also slightly increases the chance of a false positive.
                                    Spamhaus + FireHOL is a sensible default; add CrowdSec or AbuseIPDB if you want broader coverage.
                                


                                
What can never be blocked (safety belt)

                                
    
                                    
  • Loopback (127.0.0.0/8, ::1) — the host always talks to itself.
    • 
                                    
  • Private networks (10.x, 172.16/12, 192.168/16, link-local, ULA) — LAN traffic is never dropped.
    • 
                                    
  • CGNAT range (100.64/10) — covers Tailscale, Starlink, ISP-NAT users.
    • 
                                    
  • Every node in your WolfStack cluster — their addresses are stripped from the blocklist on every refresh.
    • 
                                    
  • Your own IP when you click Refresh — the API exempts the IP you’re connecting from so you can’t accidentally lock yourself out.
    • 
                                    
  • Anything you’ve allowlisted — one CIDR per line in the box above.
    • 
                                


                                
What if my own IP is on a feed?

                                

                                    A common case when renting from Hetzner, OVH, Vultr, etc.: the IP you got was previously misused by another customer
                                    and still has residual reputation. Each refresh, WolfStack checks whether any of your own cluster IPs appear
                                    on the feeds you've enabled and surfaces them in a red banner at the top of this tab if so. The banner includes:
                                

                                
    
                                    
  • Direct “check this IP” links to each provider's lookup tool, so you can verify the listing
    • 
                                    
  • Direct “submit delisting” links so you can ask the upstream feed to remove you
    • 
                                    
  • A “faster path” section with hosting-provider-specific instructions for getting a clean IP swap
    • 
                                

                                

                                    Importantly: WolfStack always exempts your own IPs from your own blocklist, so this never affects
                                    your cluster traffic. The warning matters because other networks pulling the same feeds may silently
                                    drop traffic from your server, which surfaces as “some users can't reach my service” with no obvious cause.
                                


                                
A feed false-positive blocked something my container needs

                                

                                    Add the destination CIDR to the Allowlist above and click Save. The next refresh strips that range from
                                    the active blocklist in both directions. For one-off addresses use a /32 (or /128 for IPv6).
                                


                                
Emergency pause — when to use it

                                

                                    The big ⏸ Pause enforcement button removes the kernel rule immediately while keeping
                                    everything else (config, feeds, blocklist) intact. Use it when:
                                

                                
    
                                    
  • A customer can’t reach you and you suspect threat-intel is the cause — pause first, debug second.
    • 
                                    
  • You’re running a temporary task that needs to talk to an IP that might be on a feed (security research, pentest, etc.).
    • 
                                    
  • You’re troubleshooting a network issue and want to rule threat-intel out of the picture.
    • 
                                

                                
Click ▶ Resume enforcement to bring it back — instant, no re-fetch needed.


                                
Common questions

                                
    
                                    
  • Will this break my customers? Only if their IP appears on a public threat feed. That’s rare for legitimate users; we recommend dry-run mode first to verify.
    • 
                                    
  • What about IPv6? Spamhaus dropv6 covers it. Other v6-aware feeds are rolling out gradually.
    • 
                                    
  • Does this need ipset? Yes — the kernel-level matcher uses it. Most distros have it pre-installed; if it's missing, WolfStack will try to install it automatically the first time you flip to enforce mode (apt / dnf / pacman / zypper, depending on your distro).
    • 
                                    
  • How often do feeds refresh? Whatever you set above — default 6 hours. Public feeds update at most a few times per day, polling more often is just rude.
    • 
                                    
  • Does this work in a cluster? Each node has its own threat-intel config and ipset. Cluster-wide propagation is a future enhancement.
    • 
                                    
  • Where’s the data stored? Config at /etc/wolfstack/threat-intel.json, runtime state at /etc/wolfstack/threat-intel-state.json. Both survive restarts; ipsets are restored from state on boot.
    •

What is WolfStack?

                    
WolfStack is a unified server management
                        dashboard for managing containers, VMs, networking, storage, and more across multiple servers —
                        all from a single web interface.
                        It supports both standalone Linux servers and Proxmox VE clusters.


                    
Quick Start

                    
Get your cluster up and running in
                        minutes:

                    
    
                        
  1. Install WolfStack on each server — run this on every machine you want to
                            manage:
                            
    curl -sSL https://raw.githubusercontent.com/wolfsoftwaresystemsltd/WolfStack/master/setup.sh | sudo bash
    
                        
    2. 
                        
  2. Get the token from each server — after installation, each server displays
                            its cluster token. You can also retrieve it later by running wolfstack --show-token
                            on that server.
    3. 
                        
  3. Open the web UI on one server — navigate to http://your-server-ip:8553
                            in your browser and log in. You only need to log in to one server — this will be your
                            management dashboard.
    4. 
                        
  4. Add your other nodes — click the + Add Server button in
                            the
                            sidebar and enter the address and token for each additional server or Proxmox server.
    5. 
                        
  5. Once you have added all the nodes, update WolfNet connections — go into
                            your cluster settings and click
                             Update WolfNet Connections to automatically set up peer-to-peer
                            networking between all your nodes.
                        
    6. 
                    

                    
Note:
                        You can add multiple WolfStack or Proxmox clusters to WolfStack's dashboard for a one-stop
                        server management shop.

                    

                        
 That's it! Your nodes
                            will appear in the sidebar with live metrics, and you can manage containers, networking,
                            storage, and more from a single dashboard.

Migrating a Container

                    
Migration moves a container from one node
                        to another, removing the original after successful transfer.

                    
    
                        
  1. Right-click or use the  menu on a container and select Migrate
    2. 
                        
  2. Select the Target Node from the dropdown, or choose External

    3.

Migration Progress

                    
Migration shows a detailed step-by-step
                        progress panel with five phases:

                    

                        

                            
⏹️ Stopping container — gracefully stops the running container

                            
 Creating archive — exports the container to a tar/vzdump archive

WolfStack also checks for updates automatically every 6

                        upgrade button in the banner runs the same upgrade process on the currently selected node.

                

            


            

                

                    
Troubleshooting

                    
Common issues and solutions

                

                

                    
Proxmox nodes showing offline

                    
This usually means the API token lacks
                        permissions. Check the logs:

                    
sudo journalctl -u wolfstack -f | grep -i proxmox

                    
If you see 403 Permission check failed (Sys.Audit),
                        follow the Proxmox VE Integration → Step 2 above.


                    
Ghost / duplicate nodes in sidebar

                    
WolfStack auto-cleans ghost nodes on
                        restart. If duplicates persist, restart the service:

                    
sudo systemctl restart wolfstack


                    
Remote WolfStack nodes not connecting
                    

                    
Ensure port 8553 is
                        accessible from this server and that both nodes run the same version. Check cluster secret
                        matches.


                    
Container creation fails on Proxmox node
                    

                    
If creating a container on a Proxmox node
                        fails with template errors, try refreshing the page and switching to the correct node before
                        clicking Create. The template list is cached per node — switching nodes will
                        fetch the correct templates (Proxmox pveam
                        templates vs standalone LXC templates).


                    
Proxmox containers not connecting to
                        WolfNet

                    
Proxmox LXC containers use lxcbr0 for
                        WolfNet connectivity (not vmbr0). If a
                        container can't reach WolfNet IPs:

                    
    
                        
  1. Check the container's network config: pct config <VMID>
                        
    2. 
                        
  2. Verify it uses bridge=lxcbr0
                            if WolfNet is enabled
    3. 
                        
  3. Ensure lxcbr0
                            is present: ip link show lxcbr0
                        
    4. 
                    


                    
Migration hangs or fails

                    
Migration requires network connectivity
                        between nodes. Check:

                    
    
                        
  • Both nodes can reach each other on port 8553 (or
                            configured port)
    • 
                        
  • The target node has enough disk space for the container archive
    • 
                        
  • Large containers may take several minutes — the progress panel shows elapsed time
    • 
                    


                    
Node version shows "unknown" in settings
                    

                    
This means WolfStack couldn't detect the version on the
                        remote node, usually because the node is a Proxmox host without WolfStack installed, or the API
                        proxy isn't reachable. You can still click the Install / Upgrade button to
                        install or update WolfStack on that node.
